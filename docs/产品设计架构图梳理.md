# 产品设计架构图梳理

## Agent 工具集成与执行策略

![Agent 工具集成与执行策略](./assets/agent-tool-execution-strategy.png)

### 流程概览

1. **定义智能体**：指定允许哪些 `namespace.api` 工具，以及默认审批/风险策略。
2. **读取工具 manifest**：每个工具声明 `capabilities` / `risk` / `runtime policy` / `implementation`。
3. **运行时生成「本轮可用工具集」**：
   - `agent 允许` ∩ `当前 server/worker/provider 能提供` ∩ `runtime policy 允许`
4. **把可用工具集发给大模型**，让模型选择工具。
5. **模型返回 tool call 后，server 再校验一次**：这个工具是否仍允许、参数是否合法、是否需要 approval。
6. **根据 runtime 优先级和 capabilities 选择执行实现**：
   - `server_builtin` → Server 直接执行
   - `cloud_sandbox` → `OnlyboxesProvider`
   - `local_system` → 本机 `tma-worker` / `LocalSystemProvider`
   - `auto` → 按策略选最合适实现
7. **provider / worker 执行**，回传 result、artifact refs、stdout/stderr 摘要和事件。

```mermaid
flowchart LR
  A[定义智能体] --> B[读取工具 Manifest]
  B --> C[本轮可用工具集]
  C --> D[大模型选择工具]
  D --> E[Server 二次校验]
  E --> F{执行实现}
  F --> G[server_builtin]
  F --> H[cloud_sandbox / OnlyboxesProvider]
  F --> I[local_system / tma-worker]
  F --> J[auto]
  G --> K[回传 result / artifact / events]
  H --> K
  I --> K
  J --> K
```

## cloud_sandbox：Onlyboxes 部署与接入

当执行实现选择 `cloud_sandbox` 时，底层由 **OnlyboxesProvider** 承接。Onlyboxes 不在 LobeHub / TMA 仓库内，是一套独立项目：**Console（控制面）** 负责 API、鉴权与 Worker 调度；**Worker（执行面）** 在 Docker 或 boxlite 中运行真实命令。

官方文档：

- [Install](https://onlybox.es/en/docs/install/)
- [Quick Start](https://onlybox.es/en/docs/quick-start/)
- [Console Config](https://onlybox.es/en/docs/console-config/)
- [Console API](https://onlybox.es/en/docs/console-api/)

### 架构概览

```text
┌─────────────────────────────────────────────────────────────┐
│  上游应用（LobeHub Server / TMA Worker）                      │
│  - 签发 JIT Token（HMAC-SHA256，前缀 obx_jit_v1.）            │
│  - 调用 terminalExec / terminalResource                      │
└──────────────────────────┬──────────────────────────────────┘
                           │ HTTPS
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  Onlyboxes Console（控制面）                                  │
│  - HTTP :8089  Dashboard + REST API                          │
│  - gRPC :50051 Worker 注册与任务调度                           │
│  - CONSOLE_JIT_SIGNING_KEY 校验 MCP JIT Token                │
└──────────────────────────┬──────────────────────────────────┘
                           │ gRPC
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  Onlyboxes Worker（执行面）                                   │
│  - worker-docker 或 boxlite 运行时                           │
│  - 在隔离环境中执行 Shell / Python / Node 等命令              │
└─────────────────────────────────────────────────────────────┘
```

LobeHub 通过 `OnlyboxesSandboxProvider` 对接 Console API；TMA 当前通过 `cloud_sandbox` 模式在本地 Docker 中执行命令，后续可扩展为直连 Onlyboxes Console。

### 环境要求

| 组件 | 要求 |
|------|------|
| 操作系统 | Linux 或 macOS（一键安装脚本支持） |
| 容器 | Docker + Docker Compose v2（推荐）；无 Docker 时可选 boxlite |
| 其他 | Python 3（安装脚本依赖）、systemd（一键安装用于 Worker 服务） |
| 网络 | 能访问 GitHub Releases 与 `onlybox.es`（下载安装脚本与二进制） |

### 方式一：一键安装（推荐）

适用于单机快速试用或 PoC。

```bash
curl -fsSL https://onlybox.es/install.sh | bash
```

安装脚本会自动完成：

1. 检查 Linux/macOS、Docker、Compose v2、systemd
2. 下载 compose 模板并生成初始凭证
3. 以 `docker compose up -d` 启动 Console
4. 创建 `normal` 类型 Worker
5. 下载与架构匹配的 Worker 二进制（Docker 或 boxlite）
6. 生成并启用 systemd 服务
7. 轮询直到 Worker 上线，打印访问地址与管理员账号

常用参数：

```bash
# 指定工作目录、非交互安装
curl -fsSL https://onlybox.es/install.sh | bash -s -- \
  --yes \
  --workdir /opt/onlyboxes

# 自定义 Console 端口
curl -fsSL https://onlybox.es/install.sh | bash -s -- \
  --console-http-port 18089 \
  --console-grpc-port 50052

# 固定版本
curl -fsSL https://onlybox.es/install.sh | bash -s -- --tag 0.7.1

# 无 Docker 时强制 boxlite 运行时
curl -fsSL https://onlybox.es/install.sh | bash -s -- --worker-runtime boxlite
```

安装后验证：

1. 浏览器打开 `http://127.0.0.1:8089`（或你配置的端口）
2. 使用安装输出中的管理员账号登录
3. 在 **Workers** 页面确认 Worker 状态为 **online**

### 方式二：手动安装

适合需要自定义 compose、多节点或生产环境分层的场景。

#### 1. 启动 Console

```yaml
# docker-compose.yml（示意，以官方 release 模板为准）
services:
  console:
    image: coolfan1024/onlyboxes:latest
    ports:
      - "8089:8089"
      - "50051:50051"
    environment:
      CONSOLE_HTTP_ADDR: ":8089"
      CONSOLE_GRPC_ADDR: ":50051"
      CONSOLE_HASH_KEY: "<随机生成的 HMAC 密钥>"
      CONSOLE_DASHBOARD_PASSWORD: "<管理员密码>"
      CONSOLE_JIT_SIGNING_KEY: "<与 LobeHub 共用的 JIT 密钥>"
    volumes:
      - ./db:/app/db
```

```bash
docker compose up -d
```

首次启动若无管理员账号，Console 会在日志中打印随机生成的管理员凭证。

#### 2. 创建 Worker

1. 登录 Dashboard → **Workers** → 创建 Worker
2. 记录返回的 **worker_id**、**worker_secret** 与启动命令

#### 3. 下载并启动 Worker 二进制

从 [GitHub Releases](https://github.com/onlyboxes/onlyboxes/releases) 下载对应架构的 `worker-docker`（或 boxlite 版本）：

```bash
WORKER_CONSOLE_INSECURE=true \
WORKER_CONSOLE_GRPC_TARGET=127.0.0.1:50051 \
WORKER_ID=<worker_id> \
WORKER_SECRET=<worker_secret> \
/path/to/onlyboxes-worker-docker
```

生产环境应去掉 `WORKER_CONSOLE_INSECURE=true`，并为 gRPC 配置 TLS。

#### 4. 确认 Worker 在线

Dashboard 中 Worker 状态应为 **online**，否则 LobeHub / MCP 调用会失败。

### Console 关键配置

| 变量 | 必填 | 默认值 | 说明 |
|------|------|--------|------|
| `CONSOLE_HTTP_ADDR` | 否 | `:8089` | Dashboard + REST API 监听地址 |
| `CONSOLE_GRPC_ADDR` | 否 | `:50051` | Worker 注册 gRPC 监听地址 |
| `CONSOLE_HASH_KEY` | **是** | - | Worker secret 与 access token 的 HMAC 哈希密钥 |
| `CONSOLE_DB_PATH` | 否 | `./db/onlyboxes-console.db` | SQLite 数据库路径 |
| `CONSOLE_DASHBOARD_USERNAME` | 否 | - | 首次初始化管理员用户名 |
| `CONSOLE_DASHBOARD_PASSWORD` | 否 | - | 首次初始化管理员密码 |
| `CONSOLE_JIT_SIGNING_KEY` | 接入 LobeHub 时 **是** | - | MCP JIT Token 签名密钥；未设置则拒绝 JIT 鉴权 |
| `CONSOLE_DASHBOARD_JIT_SIGNING_KEY` | 否 | - | Dashboard JIT 专用密钥，须与 `CONSOLE_JIT_SIGNING_KEY` 不同 |

**JIT 鉴权说明（LobeHub 使用 MCP JIT 通道）：**

- Token 前缀：`obx_jit_v1.`
- 签名算法：HMAC-SHA256
- 签名密钥：`CONSOLE_JIT_SIGNING_KEY`（Console 侧）与 `ONLYBOXES_JIT_SIGNING_KEY`（LobeHub 侧）**必须一致**
- Payload 需包含 `iss`（签发方）、`sub`（用户 ID）；可选 `exp`（过期时间）

轮换 `CONSOLE_JIT_SIGNING_KEY` 会使所有已签发的 MCP JIT Token 立即失效。

### 运行时镜像要求（LobeHub 对接）

LobeHub 的 `lobe-cloud-sandbox` 工具依赖 Onlyboxes Worker 暴露以下 MCP 能力：

| 能力 | 用途 |
|------|------|
| `terminalExec` | Shell 命令、代码执行、文件读写操作的兼容层 |
| `terminalResource` | 通过预签名 URL 导出沙箱内文件 |

Worker 运行镜像建议包含：

- `python3` — Python 代码执行与文件操作封装
- `node` — JavaScript 执行
- `npx` + `tsx` — TypeScript 执行
- `base64`、`find`、`grep` 等常见 Shell 工具

Onlyboxes 官方提供 LobeHub 适配镜像别名：

```text
coolfan1024/onlyboxes-runtime:lobehub
coolfan1024/onlyboxes-runtime:latest
```

### 接入 LobeHub

LobeHub 自托管部署将沙箱 Provider 从默认的 Market 切换为 Onlyboxes。

**Onlyboxes Console 侧：**

```bash
CONSOLE_JIT_SIGNING_KEY=onlyboxes-jit-signing-secret
```

**LobeHub Server 侧：**

```bash
SANDBOX_PROVIDER=onlyboxes
ONLYBOXES_BASE_URL=https://onlyboxes.example.com
ONLYBOXES_JIT_SIGNING_KEY=onlyboxes-jit-signing-secret

# 可选
ONLYBOXES_JIT_ISSUER=https://lobehub.example.com   # 默认取 APP_URL
ONLYBOXES_JIT_TTL_SEC=1800                         # JIT Token 有效期，默认 1800 秒
ONLYBOXES_LEASE_TTL_SEC=900                        # 终端会话租约，默认 900 秒
```

注意：

- `ONLYBOXES_BASE_URL` **不要**包含 `/api/v1` 路径后缀
- `ONLYBOXES_JIT_SIGNING_KEY` 必须与 Console 的 `CONSOLE_JIT_SIGNING_KEY` 完全一致
- 文件导出仍写入 LobeHub 配置的 **S3 对象存储**；若用户需要下载沙箱产物，须同时配置 S3 相关环境变量
- Agent 的 `runtimeMode` 需设为 `cloud`，才会注入 `lobe-cloud-sandbox` 内置工具集

验证步骤：

1. 创建或编辑 Agent，启用 Cloud Sandbox
2. 在对话中触发代码执行或 Shell 命令
3. 检查 LobeHub Server 日志（namespace `lobe-server:sandbox:onlyboxes`）
4. 检查 Onlyboxes Dashboard 中任务与会话状态

### 接入 TMA

TMA 当前把 `cloud_sandbox` runtime 落到 `OnlyboxesProvider`。当前实现通过本地 Docker 隔离执行工具命令：Session 第一次调用按 scope 执行 `docker run -d` 创建容器，后续调用通过 `docker exec` 复用，挂载 workspace 到 `/workspace`，并按空闲 TTL 或最大寿命回收。TMA 不自动启动 Docker daemon 或 Onlyboxes Console；Onlyboxes Console 可作为后续 Remote Provider 的执行后端。

```bash
TMA_CLOUD_SANDBOX_ROOT=.
TMA_CLOUD_SANDBOX_IMAGE=coolfan1024/onlyboxes-runtime:default
```

Session 级 `runtime_settings` 中的 `tool_runtime` / `cloud_sandbox_root` / `cloud_sandbox_image` 可覆盖上述启动默认值，无需重启服务。

| 模式 | 当前状态 |
|------|----------|
| TMA `cloud_sandbox` | 本地 Docker 隔离，不经过 Onlyboxes Console |
| LobeHub `SANDBOX_PROVIDER=onlyboxes` | 经 Onlyboxes Console API 远程执行 |
| TMA 未来 Remote Provider | 计划直连 Onlyboxes Console，与 LobeHub 共用同一套 JIT 鉴权 |

### 生产部署建议

**网络与安全：**

- Console HTTP API 应置于反向代理（Nginx / Caddy）之后，启用 HTTPS
- 限制 Console Dashboard 与 gRPC 端口的访问来源（防火墙 / 内网）
- 使用强随机值生成 `CONSOLE_HASH_KEY` 与 `CONSOLE_JIT_SIGNING_KEY`
- JIT Token 设置较短 `exp`，降低泄露风险
- Worker 与 Console 之间的 gRPC 在生产环境启用 TLS，去掉 `WORKER_CONSOLE_INSECURE`

**高可用：**

- Console：SQLite 适合单机；多实例需查阅 Onlyboxes 官方对数据库的后端支持情况
- Worker：可按负载水平扩展多个 Worker 节点，在 Dashboard 分别注册
- 为 Worker 配置资源限制（CPU、内存、磁盘），避免单任务耗尽节点

**同机 / 分机部署：**

```text
# 同机（开发 / PoC）
LobeHub Server ──localhost──▶ Onlyboxes Console (:8089)
                                    │
                                    └── Worker (127.0.0.1:50051)

# 分机（生产）
LobeHub Server ──HTTPS──▶ onlyboxes.example.com (Console)
                              │
                              ├── Worker 节点 A
                              └── Worker 节点 B
```

`ONLYBOXES_BASE_URL` 填 Console 对外可访问的 HTTPS 地址；LobeHub Server 需能访问该地址。

### 监控与排障

| 现象 | 可能原因 | 处理 |
|------|----------|------|
| LobeHub 报 `ONLYBOXES_BASE_URL and ONLYBOXES_JIT_SIGNING_KEY are required` | 环境变量未配置 | 检查 LobeHub `.env` |
| 401 / JIT 鉴权失败 | 两侧签名密钥不一致 | 核对 `CONSOLE_JIT_SIGNING_KEY` 与 `ONLYBOXES_JIT_SIGNING_KEY` |
| 命令超时 | 默认 120s | 调整命令 timeout 或检查 Worker 负载 |
| Worker offline | Worker 进程未启动或 gRPC 不可达 | 检查 systemd 服务与 `:50051` 连通性 |
| 文件导出失败 | LobeHub S3 未配置 | 配置 LobeHub 对象存储环境变量 |
| TypeScript 执行失败 | 镜像缺少 `tsx` | 换用 `onlyboxes-runtime:lobehub` 镜像 |
