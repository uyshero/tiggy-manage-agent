# 给智能体装上手脚：企业 Agent 到底该自带哪些工具？

> 从 Claude Code、Hermes、LobeHub 到 MCP，一套能真正落地的工具设计清单

大模型很聪明，但如果没有工具，它最多只能告诉你“应该怎么做”。

给它文件读取工具，它才能看见项目；给它搜索工具，它才能找到问题；给它 Shell、浏览器和代码执行能力，它才可能把任务真正完成。

但工具也不是越多越好。

工具太少，智能体只能空谈。工具太多，每轮会话先塞进几十个 JSON Schema，模型还没开始工作，上下文已经被吃掉一大块；更麻烦的是，它可能选错工具、重复调用，甚至把错误写进生产文件。

因此，企业智能体的真正能力边界，往往不取决于模型榜单，而取决于它被配了一套什么样的“手脚”。

这篇文章结合 Claude Code、Hermes Agent、LobeHub、MCP 和 TMA 的实现，回答三个问题：

1. 一个通用企业 Agent 默认应该自带哪些工具？
2. 每类工具真正难在哪里？
3. 怎样让这些工具既好用，又不失控？

## 一、先记住一个判断：工具不是 API 清单，而是一份执行合同

很多团队设计工具时，先列功能：读文件、写文件、跑命令、抓网页、开浏览器。

这只是第一步。

一个真正可用的工具，还必须同时说清楚：

- 模型什么时候能看到它；
- 哪些用户和会话可以调用；
- 什么参数需要审批；
- 执行时能访问哪些文件、网络和凭据；
- 最多返回多少内容；
- 超时、取消和失败后怎么恢复；
- 重复调用会不会产生第二次副作用；
- 最后如何证明任务真的完成了。

所以，工具体系至少要区分五个状态：

**已安装，不等于已注册；已注册，不等于本轮对模型可见；模型可见，不等于已经授权；已经授权，不等于目标环境真的可执行；执行结束，也不等于结果已经验证。**

如果这五层最终只落成一个 `enabled: true`，线上迟早会出问题。

## 二、第一阶段，先把这 5 个文件工具做好

如果从零建设企业 Agent Runtime，第一阶段不需要同时上线十几个工具。更稳的起点，是先做好五个最基础、调用频率最高的文件工具：

- `read_file`：读取一页 UTF-8 文本
- `find_files`：按路径、文件名和扩展名发现文件
- `search_files`：跨文件搜索文本内容
- `write_file`：创建或覆盖文本文件
- `edit_file`：精确修改已有文本

这里需要特别说明两个容易产生争议的设计。

### Glob 和 Grep 都需要，但不必叫 Glob 和 Grep

Coding Agent 经常需要两种完全不同的搜索：

```text
“项目中有哪些 YAML 文件？”
-> find_files

“哪些文件使用了 deprecated API？”
-> search_files
```

前者对应 Glob，搜索的是路径；后者对应 Grep，搜索的是内容。两种能力都不可缺少，但工具不必直接使用 Shell 命令名。

`find_files` 和 `search_files` 更能表达模型意图，也更容易实现跨平台路径语义、分页、权限和扫描上限。

### 不单独提供通用 binary_file 工具

大多数 Agent 没有 `read_binary_file` 或 `write_binary_file`，不是因为二进制不重要，而是因为模型真正需要的通常不是一段 Base64：

- 图片需要视觉分析；
- PDF 需要按页提取；
- Excel 需要识别 sheet 和表格；
- ZIP 需要安全解包；
- SQLite、Parquet 等格式需要专业解析库。

因此，`read_file` 负责有界检测并返回 MIME、大小、revision 和推荐处理能力；二进制正文不进入普通文本上下文。生成二进制则由 `execute_code`、格式专用工具或已有 Artifact 完成，再通过 output path 发布。

文件类型探测也不必单独做成 `file_info`。它可以作为五个文件工具共享的内部能力和统一返回元数据。

等这五个工具稳定后，再增加命令、进程、网页、人机协作、Skill、MCP 和子 Agent。默认少，能力并不一定少。正确做法是：**基础能力先做深，长尾能力按需发现。**

## 三、为什么不能默认暴露 40 个工具？

工具定义本身也会消耗上下文。

Anthropic 的 Tool Search 文档给了几个很有参考价值的数字：

- 当工具达到 10 个，或者定义超过 10k tokens，就值得评估按需加载；
- 工具数量超过 30～50 个后，模型选择正确工具的能力通常开始下降；
- 一个同时接入 GitHub、Slack、Sentry、Grafana 和 Splunk 的多服务配置，工具定义可能占用约 55k tokens；
- 按需检索通常可以减少 85% 以上的工具上下文，只加载当前任务需要的 3～5 个工具。

Hermes 也采用类似思路：核心工具直接加载，MCP 和插件工具可以延迟；当这些长尾工具的 Schema 占到模型上下文的一定比例时，再自动启用 Tool Search。

对企业 Runtime，一个更可靠的策略是把工具分成三层：

- **Hot Core**：3～5 个最高频工具，每轮可见；
- **Warm Scenario**：当前 Agent 场景需要的 5～10 个工具，会话开始时加载；
- **Cold Catalog**：MCP、插件和行业工具，搜索后再加载。

建议把常驻工具 Schema 控制在模型上下文的 8% 以内，10% 作为告警线。这个数字不是绝对标准，最终仍要用自己的模型和任务集做选择准确率评测。

工具越多越智能，是一个很常见的错觉。

真正好的工具系统，不是把所有工具摆到模型面前，而是让它在需要时快速找到正确的那个。

## 四、文件工具看起来最简单，实际上最容易出事故

“读一个文件”听起来不过是 `open + read`。

但只要进入企业场景，马上会遇到一串问题：

- 文件有几百 MB，能不能一次读完？
- 一行日志有几十 MB，按行分页还管用吗？
- byte offset 正好落在 UTF-8 中文字符中间怎么办？
- 模型读第二页时，文件已经被另一个进程修改了怎么办？
- 图片、PDF、Excel 被当成文本读，会返回什么？
- 路径指向软链接、设备文件或工作区外部怎么办？

一个合格的 `read_file`，至少应该支持三种模式：

- 小文件自动整读；
- 代码和普通文本按行读取；
- 超长行、日志和生成数据按字节读取。

每一页都要返回明确的截断状态、下一页位置和文件 revision。继续翻页时，模型必须带回同一个 revision；文件已经变化，就返回 `stale_file_revision`，要求重新读取。

否则，模型很可能把两个时间点的文件内容拼成一份“完整分析”，而系统毫无察觉。

推荐的起始配置是：

- 小文件阈值 8 KiB；
- 默认页 8 KiB 或 200 行，先到者为准；
- 单次硬上限 64 KiB；
- 行号统一使用 1-based；
- 输出明确告诉模型怎样继续。

如果已经知道关键词，不要让模型从第一页一路翻到第 300 页。`search_files` 应支持传入一个精确路径，复用单文件流式搜索，先返回匹配行号和 byte offset，再读取附近窗口。

### 路径发现和内容搜索应该拆开

`find_files` 只返回路径、类型、大小等元数据，不读取正文；`search_files` 才扫描文本内容。这样做能让两份 Schema 保持短小，也能分别设置目录数、文件数、扫描字节和结果数上限。

跨文件搜索默认跳过二进制，并明确返回跳过数量。PDF、DOCX、XLSX 内部正文搜索属于后续文档解析与索引能力，不应偷偷塞进基础 Grep 工具。

### 二进制要分流，不要塞 Base64

`read_file` 的 byte mode 是使用原始 byte offset 分页 UTF-8 文本，用来处理超长行和大型日志，不代表读取任意二进制。

遇到图片、PDF、Office、ZIP 或未知二进制时，工具应返回：

```text
文件路径
MIME / 格式
大小
revision
推荐能力
```

然后分别进入视觉、文档 Skill、`execute_code` 或受限解包流程。完整二进制通过 Object Store 和 Artifact 流转，不进入 Tool Call 和模型上下文。

### 写文件同样不能只提供一个 content 字段

`write_file` 至少要区分创建和覆盖，并支持：

- 期望文件不存在；
- 期望文件仍是某个 revision；
- 内容摘要和幂等键；
- 临时文件写入后原子替换；
- 磁盘配额和最大内容限制。

大文件也不应该让模型一次生成。更稳的方式是先写骨架和唯一占位符，再分段替换，每段完成后校验，最后确认占位符已经清零并运行测试。

文件工具真正难的地方，不是“能读写”，而是**能在并发、超大内容、编码和失败场景下保持一致**。

## 五、专用工具优于万能 Shell，但 Shell 不能消失

只给智能体一个 Bash，确实非常灵活。

它可以用 `cat` 读文件、`grep` 搜内容、`sed` 改文件、`curl` 抓网页，还能顺手执行任意程序。

问题也很明显：难审计、难限流、难做参数级审批、难得到结构化结果。

因此，文件读取、搜索、精确编辑和网页抓取应该优先使用专用工具。Shell 退回到它真正擅长的场景：构建、测试、系统诊断和长尾 CLI。

但完全禁止 Shell 也不现实。那只会逼着团队在专用工具里重新造半个终端。

更合理的设计是把 Shell 变成受约束的兜底执行面：

- 命令和参数分开传递，默认不经过 `sh -c`；
- cwd 必须在允许目录内；
- 环境变量使用 allowlist，凭据只注入引用；
- stdout 和 stderr 分开限流；
- 超时后杀死整个进程组；
- CPU、内存、PID、磁盘和网络都有硬限制；
- 后台任务交给独立 `process` 工具管理。

这里还有一个关键点：**权限审批和运行时沙箱不是一回事。**

审批回答“这条命令能不能开始执行”，沙箱回答“执行以后最多能碰到哪些文件和网络”。Claude Code 当前也是把权限规则和 Bash 的 OS 级沙箱分成两层。

只做审批，获准进程及其子进程仍可能做出意料之外的动作；只做沙箱，模型也可能在沙箱允许范围内误删文件。

两层都要有。

## 六、搜索、抓取和浏览器，不应该混成一个“上网工具”

网络能力建议明确分工：

- 搜索只返回标题、摘要、URL 和发布时间；
- 静态网页由 `web_fetch` 提取正文；
- 只有登录、点击、输入和动态渲染才使用浏览器。

抓取器必须防 SSRF。每一次请求和重定向都要检查目标地址，阻断 localhost、内网地址、云 metadata 和危险 URL scheme；还要限制下载大小、压缩比例、响应时间和正文长度。

浏览器则建议先保持精简：open、snapshot、click、type、screenshot 五个核心动作已经能覆盖大量流程。

点击和输入必须基于 snapshot 返回的元素引用，而不是让模型凭感觉猜 CSS。元素引用还要绑定页面版本，页面变化后应返回 `stale_element_ref`。

更重要的是，“点击”不能统一视为普通 write：

- 展开下拉框，只是页面内部状态；
- 提交表单、发消息、授权、删除和付款，会产生真实外部副作用。

审批策略必须看到动作的目标和后果，而不只是看到一个 `browser_click` 工具名。

## 七、图片、PDF 和 Office，不要假装它们是文本文件

图片、PDF、DOCX、XLSX 和 PPTX 应该进入不同的媒体处理路径。

图片交给视觉模型；PDF 按页提取文本和渲染；Excel 返回 sheet、表头、样例和计算结果；PPT 返回 slide 文本、备注和页面预览。

几个基本原则：

- 原始二进制不进入普通文本结果；
- 解析内容必须保留页码、sheet 或 slide 定位；
- 大型结果写入 Artifact，只给模型摘要和引用；
- 宏、嵌入对象、外链和压缩包内容按不可信输入处理。

“binary read failed”不是一个合格的产品体验。即使当前没有解析器，也应该明确告诉模型下一步该调用哪个文档或视觉工具。

## 八、MCP 和 Skill 是扩展层，不是权限后门

MCP 让 Agent 可以动态接入数据库、GitHub、Notion、企业内部系统和各种 SaaS。

但 MCP Server 提供的工具名称、描述、annotations 和结果，都属于远端输入，不能因为它自称 `readOnly` 就直接相信。

企业 Runtime 仍然要做：

- 校验输入和输出 Schema；
- 套用 workspace 和用户权限；
- 限制超时、并发和调用频率；
- 防止 token passthrough 和 SSRF；
- 记录审计，但不泄漏凭据和远端正文；
- 工具目录发生变化时重新索引，但不能静默扩大当前会话权限。

Skill 也是一样。

Skill 本质上是操作说明和流程注入，不是“安装后自动获得所有权限”。Skill 中引用的脚本、文件和工具，执行时仍然要经过普通的工具授权与沙箱。

安装 Skill 还应检查来源、版本、许可证、可执行文件、宏、压缩包、软链接和供应链证明。

扩展能力可以很多，但权限边界不能随扩展一起失控。

## 九、子 Agent 的价值是隔离上下文，不是无限套娃

子 Agent 最适合处理三类任务：

- 搜索和日志很多，但父会话只需要结论；
- 几个独立工作流可以并行；
- 需要不同模型、提示词或工具权限。

Claude Code 的子 Agent 会运行在独立上下文中，并可以限制工具、模型、最大轮次和 worktree。Hermes 也通过 delegate 处理隔离任务。

企业实现里还应加上硬约束：

- 递归深度 2～3；
- 单个父任务最多并行 3～5 个；
- 子 Agent 默认移除人机提问和再次委托；
- 文件写入使用独立工作区或明确 ownership；
- 每个子任务都有最大轮次、token 和费用；
- 父 Agent 只回收摘要、结构化结果、证据和 Artifact。

把子 Agent 的完整上下文再灌回父会话，就失去了隔离的意义。

## 十、所有工具上线前，都过这 8 个问题

不管是读文件、开浏览器还是调企业 API，每个工具都应该回答：

1. 输入字段、默认值和互斥关系是什么？
2. 模型、UI 和程序分别看到什么输出？
3. 默认上限和硬上限是多少？
4. 截断以后如何继续，cursor 和 revision 是否稳定？
5. 有哪些稳定错误码，模型下一步该怎么做？
6. 路径、网络、命令、凭据和租户权限如何强制？
7. 相同调用重复一次会发生什么？
8. 如何测试和证明工具真的完成了任务？

如果其中任何一项只能回答“靠模型自己注意”，这件工具就还没有设计完。

## 十一、从零落地，建议按这个顺序

第一阶段，先完成文件系统五件套：读、找、搜、写、改。把分页、revision、UTF-8、路径守卫、二进制分流和原子写做对。

以本文对应项目为例，这五件套已经完成核心实现：旧单文件搜索保留兼容但从默认模型工具面隐藏；二进制只返回类型和路由元数据；本地、沙箱与 Worker 使用同一契约。上线前剩下的重点不再是继续加 API，而是灰度、延迟与截断指标、真实仓库压测，以及模型能否稳定区分“找文件”和“搜内容”的行为评测。

第二阶段，加入命令、代码执行和进程管理。补齐超时、取消、后台日志、资源配额和 Artifact。

第三阶段，加入网页搜索和抓取。先把 SSRF、重定向、下载上限和不可信内容标记做好。

第四阶段，加入人机交互、计划和完成证据，让长任务可以暂停、恢复和验证。

第五阶段，建设 Tool Search。将工具分成 Hot、Warm、Cold 三层，控制 Schema 对上下文的侵占。

第六阶段，再根据业务增加浏览器、文档、Skills、MCP 和子 Agent。

每一阶段都要同时交付契约测试、权限测试、故障注入和模型行为评测。不要等工具全部开发完，才开始补安全和可观测性。

## 结语

智能体自带工具，本质上是在给模型一套受约束的手脚。

读，是为了感知，但不能一次吞掉整个世界；

写与执行，是为了行动，但不能绕过权限和沙箱；

搜索与浏览器，是为了补全环境，但不能变成无底洞；

人机协作与计划，是为了管理不确定性，而不是让模型把所有判断都甩给用户；

Skill、MCP 和子 Agent，是为了扩展能力，但不能顺便扩大权限。

好的工具集，不是功能清单的堆叠，而是一份把成本、安全、可恢复性和证据都算清楚的合同。

合同清楚，模型才像真正的同事。

合同含糊，模型就只是一个会调用 API 的高成本脚本。
